# Marketo 통합 우수 사례

Marketo 인스턴스의 공유 API 제한에 대한 통합 디자인을 참조하십시오. 일괄 처리, 캐싱 및 일관된 요청 속도를 사용하여 처리량과 안정성을 개선합니다.

## API 제한

- **일일 할당량:** 대부분의 구독에는 하루에 50,000개의 API 호출이 할당됩니다. 할당량은 매일 오전 12:00(CST)에 재설정됩니다. 일일 할당량을 늘리려면 계정 관리자에게 문의하십시오.
- **속도 제한:** 각 인스턴스는 20초당 100개의 API 호출로 제한됩니다.
- **동시 실행 제한:** 각 인스턴스는 최대 10개의 동시 API 호출을 허용합니다.
- **일괄 처리 크기:** 리드 DB는 300개의 레코드를 지원하며 자산 쿼리는 200개의 레코드를 지원합니다.
- **REST API 페이로드 크기:** 1MB.
- **일괄 가져오기 파일 크기:** 10MB.
- **SOAP 최대 일괄 처리 크기:**&#x200B;개의 레코드.
- **일괄 추출 작업:** 실행 중인 두 작업 및 대기 중인 열 작업(포함).

## 빠른 팁

- 응용 프로그램이 할당량, 비율 및 동시 사용 리소스를 다른 응용 프로그램과 공유하므로 보존적 사용 제한을 설정하십시오.
- 가능한 경우 Marketo 대량 및 일괄 처리 메서드를 사용합니다. 필요한 경우에만 단일 레코드 또는 단일 결과 호출을 사용합니다.
- 환율 또는 동시성 제한으로 인해 실패한 API 호출을 다시 시도하려면 [지수 백오프](https://en.wikipedia.org/wiki/Exponential_backoff)를 사용하십시오.
- 사용 사례에 도움이 되지 않는 한 동시 API 호출을 피하십시오.

## 일괄 처리

삽입 및 업데이트의 경우 가능한 한 적은 수의 트랜잭션으로 레코드를 그룹화합니다. 데이터 저장소에서 레코드를 검색할 때, 각 변경 사항에 대해 하나의 요청을 제출하는 대신 제출 전에 집계하십시오.

## 허용 가능한 지연

통합을 디자인할 때 허용 가능한 지연(API 호출을 제출하기 전의 최대 시간)을 정의합니다. 이 선택은 사용 사례에 맞는 Marketo 메서드 및 구성 옵션을 결정합니다.

예를 들어, 사용자가 체험판을 시작할 때 영업사원에게 알리는 실시간 통합은 즉각적인 후속 조치가 필요할 때 1개의 일괄 처리를 제출할 수 있습니다. 대부분의 사용 사례에서는 호출을 대기열에 넣고 일괄 처리하여 더 많은 지연을 허용하고 보다 효율적으로 운영할 수 있습니다.

| 허용 가능한 지연 | 기본 메서드 | 참고 |
| --- | --- | --- |
| 낮음(&lt;10s) | 동기 API(일괄 처리 또는 일괄 처리 해제) | 사용 사례에서 이를 요구하는지 확인합니다. 대용량 사용 사례에 대해 즉시 및 동기 호출을 전송하면 일일 API 할당량이 빠르게 소모되고, 잠재적으로 요율 및 동시 실행 제한을 모두 초과할 수 있습니다. |
| Medium(10s - 60m) | 동기 API(일괄 처리) | Marketo에 대한 인바운드 데이터 통합의 경우 연령 및 크기 제한이 모두 있는 큐를 사용하는 것이 좋습니다. 두 제한에 도달하면 큐를 플러시하고 누적된 레코드로 API 요청을 제출합니다. 이는 큐의 사용 기간이 허용하는 만큼 많은 레코드를 일괄 처리하는 동시에, 필요한 케이던스로 요청이 발생하도록 하며 속도와 효율성 간의 강력한 타협입니다. |
| 높음(>60m) | 대량 가져오기/내보내기(지원되는 경우) | 인바운드 데이터 통합을 위해 가능하면 Marketo Bulk API를 통해 레코드를 큐에 추가하고 제출해야 합니다. |

## 일별 제한

각 API 활성화 Marketo 인스턴스에는 최소 10,000개의 REST API 호출이 매일 할당되어 있지만, 50,000개 이상은 일반적입니다. 각 인스턴스에는 500MB 이상의 대량 추출 용량도 있습니다. 일일 추가 용량은 Marketo 구독의 일부로 구매할 수 있지만, 애플리케이션 디자인은 일반적인 구독 한도를 고려해야 합니다.

용량은 인스턴스의 모든 API 서비스 및 사용자가 공유합니다. 중복 호출 및 배치 기록을 가능한 한 적은 수의 호출로 제거합니다.

호출 효율이 가장 높은 가져오기 방법은 [잠재 고객/사용자](https://developer.adobe.com/marketo-apis/api/mapi#tag/Bulk-Import-Leads/operation/importLeadUsingPOST) 및 [사용자 지정 개체](https://developer.adobe.com/marketo-apis/api/mapi#tag/Snippets/operation/createSnippetUsingPOST)에 사용할 수 있는 Marketo 대량 가져오기 API입니다. Marketo은 [리드](bulk-lead-extract.md) 및 [활동](bulk-activity-extract.md)에 대한 대량 추출도 제공합니다.

### 캐싱

다음 작업의 결과는 일반적으로 드물게 변경되므로 하루 이상 클라이언트측에 캐시될 수 있습니다.

- 설명 작업 결과
- [활동 유형](https://developer.adobe.com/marketo-apis/api/mapi#tag/Activities/operation/getAllActivityTypesUsingGET)
- [파티션](https://developer.adobe.com/marketo-apis/api/mapi#tag/Leads/operation/getLeadPartitionsUsingGET)

리드 또는 활동 데이터 보강과 같은 사용 사례의 경우 프로그램, 이메일 및 폴더와 같은 에셋 유형을 캐시할 수도 있습니다.

## 비율 제한

각 Marketo 인스턴스에는 모든 타사 API 서비스에 대해 20초당 100개의 호출 공유 비율 제한이 있습니다. 호출이 이 제한을 초과하는 경우 API는 606 오류 코드를 반환합니다.

일반적으로 여러 API 통합 및 사용자가 사용 가능한 용량을 공유할 수 있도록 각 타사 통합을 20초 이내에서 50회 호출로 제한합니다. 일부 사용 사례에는 전체 제한이 필요할 수 있습니다. 그러나 일괄 처리를 사용하고 낮은 처리량을 목표로 하는 응용 프로그램은 일반적으로 응답 속도가 더 높고 일관성이 유지되며 지연 시간이 거의 증가하지 않습니다.

## 동시 실행 제한

각 Marketo 인스턴스에는 동시에 실행되는 REST API 호출의 공유 제한이 10개입니다. 애플리케이션이 이 제한의 유일한 소비자라고 가정하지 마십시오.

Marketo은 처리 중이며 아직 반환되지 않은 호출을 계산합니다. 호출이 반환되면 더 이상 동시성 한도에 계산되지 않습니다.

대부분의 통합은 동시 호출의 이점을 받지 않습니다. 동시성을 구현하는 경우 처음에는 애플리케이션을 5개 이하의 동시 요청으로 제한합니다. 응용 프로그램에 더 많은 내용이 필요하다고 결정한 후에만 제한을 늘리십시오.

## 오류수

드문 경우를 제외하고 API 요청은 HTTP 상태 코드 200을 반환합니다. 비즈니스 논리 오류도 200을 반환하지만 응답 본문에 세부 정보를 포함합니다. 자세한 내용은 [오류 코드](error-codes.md)를 참조하십시오.

HTTP 이유 구는 선택 사항이며 변경될 수 있으므로 평가하지 마십시오.
